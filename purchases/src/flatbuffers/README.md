# FlatBuffers PoC

Proof of concept for passing a discrete **section** of a backend HTTP response to the SDK as a
[FlatBuffer](https://flatbuffers.dev/) instead of JSON. This is an evaluation spike, not a wired
feature: nothing in the production request/response path is changed, and the parser is exercised
only by unit tests.

## Data flow

```
Backend                                   SDK
-------                                   ---
build FlatBuffer (FlatBufferBuilder)      HTTPResult.body : JSONObject   (existing, unchanged)
  -> base64 encode                          -> read "payload_fb" field
  -> put in JSON field "payload_fb"          -> base64 decode
                                              -> Payload.getRootAsPayload(bytes)
JSON: { ..., "payload_fb": "<b64>" }         -> map to PayloadData (plain Kotlin)
```

The FlatBuffer rides as a base64 string inside the normal JSON body, so the transport
(`HTTPClient`, `HTTPResult`, the ETag cache) needs no changes. The SDK reads one extra field from
the already-parsed `JSONObject` and decodes it.

## Files

- `schema/payload.fbs` — the mock schema: a single `Payload` table with a `config` byte array and
  a `blobs` "map" of bytes to bytes. FlatBuffers has no native map type, so the map is modeled as
  a vector of key/value entries (`Blob { key:[ubyte]; value:[ubyte]; }`).
- `../main/kotlin/com/revenuecat/purchases/flatbuffers/generated/` — **reader-only** Kotlin
  accessors generated by `flatc` (committed; do not hand-edit). The regen script strips the writer
  helpers (everything taking a `FlatBufferBuilder`) and marks the classes `internal`. `internal`
  keeps them out of the public API and exempts them from `-Xexplicit-api=strict`. They bind to the
  vendored runtime below (same package), so the shipped SDK pulls in no `com.google.flatbuffers`.
- `../main/java/com/revenuecat/purchases/flatbuffers/generated/` — the **vendored** read-only
  FlatBuffers Java runtime (`Table`, `Constants`, `Utf8`/`Utf8Safe`/`Utf8Old`), copied from
  flatbuffers-java 25.2.10 (Apache-2.0), repackaged into this package and made **package-private**
  so it neither leaks into the public API (Metalava emits only public/protected) nor collides with
  a consumer's own FlatBuffers. See "Shipping no FlatBuffers dependency" below.
- `../main/kotlin/com/revenuecat/purchases/flatbuffers/FlatBuffersPayloadParser.kt` — the
  SDK-side parser. Decodes the section and returns the domain model, or `null` on any error
  (same swallow-and-log behavior as `OfferingParser`).
- `../main/kotlin/com/revenuecat/purchases/flatbuffers/PayloadData.kt` — plain Kotlin domain
  models (`PayloadData`, `BlobEntry`). The `blobs` map is exposed as a list of key/value entries
  because `ByteArray` is unsound as a `Map` key (identity equality). Generated types never leak
  past the parser.
- `../test/java/com/revenuecat/purchases/flatbuffers/testgen/` — the **full** read+write accessors
  (generated alongside the reader-only copy), bound to the real `com.google.flatbuffers` runtime
  (a test-only dependency). The encoder fixture (the "backend side") builds buffers with these.
- `../test/java/com/revenuecat/purchases/flatbuffers/` — the encoder fixture and a roundtrip test.

## Shipping no FlatBuffers dependency

The SDK ships **no** `com.google.flatbuffers` code and declares no FlatBuffers runtime dependency,
so it cannot conflict with a consumer that already uses FlatBuffers at a different version. This is
achieved without shading (no Shadow plugin here, and AGP does not relocate AAR classes reliably):

- **Decoding** needs only a tiny slice of the runtime — the `Table` base class plus `Constants`
  and the `Utf8*` helpers it transitively compiles against. Those 5 files are vendored under
  `src/main/java/.../flatbuffers/generated`, repackaged into that package and made package-private.
  Package-private keeps them out of the tracked public API (Metalava emits only public/protected)
  and the unique package means no collision with a consumer's `com.google.flatbuffers`.
- **The shipped generated accessors are reader-only.** flatc's writer helpers all take a
  `FlatBufferBuilder`, which the SDK never needs; the regen script strips them, so nothing in
  `src/main` references `FlatBufferBuilder`.
- **Encoding (the backend side) stays test-only.** `flatbuffers-java` is a `testImplementation`
  dependency and the full read+write accessors live in `src/test/.../testgen`. The roundtrip test
  encodes with the real runtime and decodes with the vendored one, so it doubles as a
  wire-compatibility check between the two.

If the schema ever needed strings or other features, extend the vendored slice accordingly (or
reconsider shading); for this PoC the read path is deliberately minimal.

## Regenerating the accessors

```bash
./scripts/generate-flatbuffers.sh
```

This runs `flatc` once and post-processes the output into two flavors: the reader-only
`src/main/.../generated` copy and the full `src/test/.../testgen` copy (see
`scripts/flatbuffers_postprocess.py`). The **vendored Java runtime** under
`src/main/java/.../generated` is maintained by hand and is *not* regenerated.

**Version match is mandatory.** Generated code pins the runtime version via
`Constants.FLATBUFFERS_<version>`. The `flatc` compiler version, the vendored runtime, and the
test-only `flatbuffers` dependency in `gradle/libs.versions.toml` (currently `25.2.10`) must all be
the same released version, otherwise the generated Kotlin will not compile. The script enforces the
flatc/version match. Note that Homebrew may ship a newer `flatc` than the latest published
`com.google.flatbuffers:flatbuffers-java` on Maven Central; pin all three to the same release.

## JSON vs FlatBuffers (evaluation notes)

- **Zero-copy reads:** fields are read directly from the byte buffer with no intermediate object
  graph, unlike `JSONObject`/`kotlinx.serialization` which allocate during parse.
- **Size:** binary is typically smaller on the wire than equivalent JSON (no field-name
  repetition), though base64 embedding adds ~33% back; a fully binary response body would avoid
  that (see below).
- **Schema evolution:** new fields can be appended to a table and old readers ignore them; this
  matches our existing tolerance for unknown JSON keys.
- **Cost:** a `.fbs` schema plus a codegen step, the version-match constraint above, and (because
  we ship no dependency) maintaining the small vendored read-only runtime described in "Shipping no
  FlatBuffers dependency".

## Not done here (follow-ups)

- Wiring into a real endpoint (e.g. a section of the offerings response in `Backend.kt`).
- A **fully binary response body** (binary `Content-Type` instead of a base64 JSON field). That
  is more efficient but more invasive: `HTTPResult.payload` is a `String` today, so it would
  require retaining raw bytes through `HTTPClient` and the ETag cache.
